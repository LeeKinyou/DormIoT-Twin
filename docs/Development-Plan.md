# DormIoT-Twin 敏捷开发全流程规划

> 基于 Scrum 框架 · 1 周 Sprint · 6 轮迭代交付

---

## 1. 敏捷方法论

### 1.1 框架选择

采用 **Scrum** 轻量化实践，适配课程设计 / 小型团队（2-4 人）节奏：

| 要素 | 设定 |
|------|------|
| Sprint 周期 | **1 周**（周一计划 → 周五评审） |
| 总迭代数 | **6 个 Sprint**（共 6 周） |
| 每日站会 | 15 分钟，同步阻塞项 |
| 评审会议 | Sprint 最后一天，演示可工作软件 |
| 回顾会议 | 评审后 30 分钟，改进流程 |

### 1.2 角色定义

| 角色 | 职责 |
|------|------|
| **产品负责人（PO）** | 管理 Product Backlog、定义验收标准、确认 Sprint 交付物 |
| **Scrum Master（SM）** | 移除阻塞、主持会议、保护团队免受干扰 |
| **开发团队（Dev）** | 领取任务、编码实现、单元测试、代码审查 |

### 1.3 Sprint 流程

```
周一                周二-周四              周五
┌──────────┐      ┌──────────────┐      ┌──────────────┐
│ Sprint   │      │ 每日开发      │      │ Sprint 评审   │
│ 计划会议  │ ──▶  │ + 每日站会    │ ──▶  │ + 回顾会议    │
│ (2h)     │      │ (15min/天)   │      │ (1.5h)       │
└──────────┘      └──────────────┘      └──────────────┘
  选取故事            编码 + 测试           演示 + 反思
  拆分任务            代码审查              改进项记录
```

## 4. 开发工作流

### 4.1 Git 分支策略

```
main ─────────────────────────────────────────────────▶ 稳定发布
  │
  ├── feature/S1-scaffold ────────────── PR ──▶ merge
  ├── feature/S2-simulation ──────────── PR ──▶ merge
  ├── feature/S3-mqtt ────────────────── PR ──▶ merge
  ├── feature/S4-storage ─────────────── PR ──▶ merge
  ├── feature/S5-ui ──────────────────── PR ──▶ merge
  └── feature/S6-validation ──────────── PR ──▶ merge
```

- **main**：始终可部署，只接受 PR 合并
- **feature/S{n}-{name}**：每个 Sprint 一个特性分支
- **hotfix/**：线上紧急修复

### 4.2 Commit 规范

```
<type>(<scope>): <subject>

类型：
  feat      新功能
  fix       Bug 修复
  test      测试相关
  refactor  重构
  docs      文档
  chore     构建 / 依赖 / 配置

示例：
  feat(simulation): 实现 VirtualIoTDevice 高斯噪声仿真
  fix(rule_engine): 修复冷却时间判断逻辑
  test(storage): 补充 Redis 缓存单元测试
```

### 4.3 PR 合并检查清单

- [ ] 代码通过 `uv run pytest` 全部测试
- [ ] 新增功能有对应单元测试
- [ ] 类型注解完整（`mypy` 无 error）
- [ ] Commit message 符合规范
- [ ] 至少 1 人 Code Review 通过
- [ ] 无硬编码密码 / token

```
         └──────────┘
```

### 5.2 各层测试覆盖

## 6. Definition of Done（DoD）

每个用户故事必须满足以下条件才能标记为完成：

| 检查项 | 标准 |
|--------|------|
| **功能实现** | 验收标准全部满足 |

| **Code Review** | 至少 1 人审查通过 |
| **文档更新** | 接口变更同步更新 PRD |

每个 Sprint 必须满足：

| 检查项 | 标准 |
|--------|------|

| **无技术债** | 无 TODO/HACK 残留（或已记录到 Backlog） |

## 7. 风险清单与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| EMQX Docker 镜像拉取慢 | 中 | 阻塞 S3 | 提前预拉取镜像；备用方案：使用公共 MQTT 测试服务器 |
| Streamlit `st.fragment` 实时性不足 | 中 | 影响 UI 体验 | 备选方案：`st_autorefresh` 第三方组件；降低刷新频率到 5s |
| SVG 楼层图交互方案不可行 | 低 | 阻塞 S5 | 备选：用 Streamlit Columns 网格布局替代 SVG |
| 50 节点并发 MQTT 丢消息 | 低 | 影响数据完整性 | QoS=1 确保至少一次投递；增加消息计数验证 |
| MySQL 连接池耗尽 | 低 | 告警写入失败 | 使用 SQLAlchemy 连接池 + 重试机制 |
